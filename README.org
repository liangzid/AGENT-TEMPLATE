* What Is This?

AGENT-TEMPLATE is a reusable configuration template for AI coding agents. It provides a battle-tested set of agent instructions (prompts) and permission configurations, enabling tools like Claude Code or [[https://github.com/pi-sdk/pi][PI Coding Agent]] to assist you in software development and academic writing in a disciplined, efficient, and safe manner.

Recommended agent: PI Coding Agent.

The core philosophy: *prompt reusability outweighs the value of any single code output*. Rather than tweaking agent behavior on the fly, maintain a stable, general-purpose configuration template.

* Core Design of AGENTS.md

** Conversation Rules

The rules in AGENTS.md take priority over everything else. Key principles:

1. *Disagree when you disagree*. If the user's premise is wrong, the agent must point it out rather than appease.
2. *Never fabricate*. When uncertain, read files, run commands, or search. If still unsure, say "No evidence to show..." or "I do not know that."
3. *Stop when confused*. If instructions are ambiguous, ask for clarification. Never proceed silently.
4. *Minimal code changes*. No drive-by refactors, reformatting, or unrelated modifications.
5. *Grilling*. During planning, the agent must interview the user relentlessly about every design branch, one question at a time, until all dependencies are resolved.
6. *Record discussions*. Every 5 turns, briefly record key decisions into =./records/[Task].org=. Remove noise and dead ends; keep only essential conclusions.

** Coding Rules

- No self-commit (=git commit=). The user decides when to commit.
- No single function should be too long or too large.
- *Never mock data*, even during debugging.
- Clarity-first naming. E.g., =maio2025.py= → =loading_miao2025.py=.
- Use =uv= for Python project management.

** Code Review Requirements

Every source file must carry the following header:

#+begin_example
======================================================================
`(upcase (file-name-nondirectory (file-name-sans-extension (buffer-file-name))))`

1. (What-this-file-aims-to-do)
2. (The-calling-chains-or-graphs-of-each-function-or-components)
3. (Modification-history-and-reason)

    Author: Zi Liang <zi1415926.liang@connect.polyu.hk>
    Copyright © `(format-time-string "%Y")`, Zi Liang, all rights reserved.
    Created: `(format-time-string "%e %B %Y")`
======================================================================
#+end_example

Every function and critical point must also be commented. Explicitly mark items that need user review.

** Academic Writing Rules

- Always use LaTeX, never Markdown.
- Use =$$= or =\begin{...}\end{...}= for math. Never use =\(\=).
- Forbidden words: solely, leverage, delve into, tapestry, furthermore.
- Forbidden punctuation: em dash (—), double em dash (--), and semicolons (;).
- Never = \textbf{} = the first sentence inside an enumerate environment.
- No contractions (e.g., it's → it is). Minimize ='s= constructions.
- Respect existing abbreviations in context; do not re-expand them.
- Preserve any formatting adjustments made by the user (团长).


* Design Philosophy

- *Write documents, not chat messages*. Use documents to communicate requirements. Be thorough.
- *No SDD (Specification-Driven Development overkill)*. Keep processes practical.
- *Humans must read, review, and ask*. Documents are for human consumption; do not delegate understanding to the agent.
- *Prompt language does not matter*. English is most convenient, but Chinese is equally fine. Use whatever flows naturally.
- *Reusability comes first*. A good prompt template serves countless tasks. Its value far exceeds any single code output.
